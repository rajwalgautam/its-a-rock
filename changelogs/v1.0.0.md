# v1.0.0

It's A Rock reaches 1.0 — with a batch of fixes from real-world use and a
versioning correction.

## What's new

**Fixes**
- **Location picker works end to end** — tap the dropdown arrow to browse your
  saved locations, tap one to select it (taps no longer get dropped), and the
  arrow now gives way to a clear (✕) button as soon as you start typing.
- **Simpler grade ranges** — the softer / even / harder buttons no longer show
  when you pick a range. Ranges are plain, like "V4 – V5".
- **The keyboard no longer hides the notes field** — the add and edit forms now
  keep whatever you're typing in view above the keyboard.
- **Same-day climbs validate correctly** — logging a climb you started and sent
  on the same day no longer triggers a false "start date can't be after the send
  date" error.

**Version**
- The previous build shipped showing its version as "v0.50" instead of "v0.5.0",
  which also broke the in-app update check so it stopped seeing newer releases.
  This release corrects the version to 1.0.0 and restores update detection, so
  "Check for Updates" works again. Updating preserves all your climbs, photos,
  and locations.

## Under the hood

- Date validation compares by **calendar day** (`startOfDayMs`) instead of raw
  timestamps, and the form stores dates normalized to local midnight — so a
  same-day start/send is always valid (#90).
- `GradePicker` gained a `showModifiers` flag; range endpoints render base-only
  and still serialize as `V4-V5`, so no schema or data change is needed (#88).
- `LocationPickerField` rebuilt: the chevron toggles a browse-all list of saved
  gyms, suggestions are plain `Pressable`s inside a keyboard-persisting
  `ScrollView` (fixes dropped taps), and the trailing control switches between
  chevron (empty) and clear (typing) (#86).
- Both the add and edit forms use `KeyboardAvoidingView`, the edit screen now
  persists taps, and Android `softwareKeyboardLayoutMode` is set to `pan` so the
  focused field stays visible (#89).
- The release workflow now requires a matching `changelogs/vX.Y.Z.md` (leading
  `v` included) **before** creating the tag, so a malformed version like "v0.50"
  fails fast instead of shipping (#94). `isNewerVersion` parses versions into
  numeric segments, which is why the malformed "0.50" had outranked every real
  "0.x.y".
